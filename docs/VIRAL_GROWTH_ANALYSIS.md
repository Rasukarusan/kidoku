# Kidoku バイラル成長分析 — バズるために必要な機能と運用

> 2026-07 時点のコードベース調査に基づく分析。現状機能の棚卸し → バイラルループのギャップ分析 → 優先度付き機能ロードマップ → 運用施策の順に整理する。

## TL;DR

Kidoku には「バズの種」となる資産（AI読書性格診断・Wrapped風OG画像・公開本棚・匿名いいね）が既に揃っている。一方で、**シェアの出口が X のリンク共有のみ**、**流入者が登録せずに体験できる入口がない**、**再訪を促す仕組み（メール・プッシュ・ストリーク）が未接続**という3点でバイラルループが途切れている。最優先は「①シェア画像の直接共有（Instagram対応）」「②未登録お試しAI診断」「③計測基盤」の3つ。

## 1. 現状の強み（すでにあるバイラル資産）

| 資産 | 実装箇所 | バイラル観点での価値 |
|---|---|---|
| AI読書性格診断（「一言でいうとこんな人」等5項目） | `api/ai-summary/_create.ts`, `AiSummaries/` | 診断系コンテンツはSNS拡散の王道。OGカード（`type=ai`）はコード内で「拡散向け」と明記済み |
| Wrapped風 年間読書まとめOGカード | `api/og.tsx`（`type=wrapped`） | Spotify Wrapped型の季節性バズの土台 |
| 動的OG画像生成（book / year / ai / wrapped の4種） | `api/og.tsx`（@vercel/og, 1200×630） | リンクシェア時の見栄えは確保済み |
| 公開本棚 + ISR + サイトマップ | `/[user]/sheets/[year]`, `/books/[bookId]`, `sitemap.xml.tsx` | 全ページ公開・SEO流入の受け皿になる |
| 匿名いいね | `Like.anonymousId`, `LikeButton.tsx` | 未登録者でも反応できる＝流入者の初回アクションの摩擦が極小 |
| フォロー / コメント / 通知 / フィード | `features/social/`, `/discover`, `/notifications` | ソーシャルグラフの基盤は完成している |
| 横断検索（MeiliSearch）・人気本・読了数ランキング | `/search`, `/discover` | 発見面（ディスカバリー）の骨格あり |
| バーコードスキャン登録 | `EnhancedBarcodeScanner.tsx` | モバイルでの登録摩擦が低い |
| X共有 + Web Share API | `utils/socialShare.ts`（`#kidoku` 付き） | 最低限のシェア導線 |

## 2. バイラルループのギャップ分析

バイラル成長は「**コンテンツ生成 → 共有 → 流入 → 登録・体験 → 継続 → 再共有**」のループが1周して初めて成立する。各段で現状を評価する。

### 2-1. 共有（出口）— ★最大のボトルネック

- **シェア手段が X のリンク共有と Web Share のみ**。読書系ユーザーが多い Instagram（ストーリーズ）・Threads・LINE への導線がない。
- **OG画像は「リンクプレビュー」でしか使われていない**。画像そのものをダウンロード／端末に保存して投稿する導線がなく、Instagram Stories（9:16）用の縦型カードも存在しない。診断・Wrapped系のバズは「画像を貼る」文化圏で起きる。
- **シェアのトリガーが少ない**。共有ボタンは「本の感想モーダル」と「AI診断結果」の2箇所のみ。最も感情が動く瞬間＝**読了登録の直後**にシェア提案がない。
- Wrapped は通年ページの一部で、**「年末に自動生成されて全員に届く」体験になっていない**（Spotify Wrapped のバズは"全員に同時に届く"ことで発生する）。
- シェアURLに UTM 等のパラメータがなく、**どのシェアが流入を生んだか計測不能**。

### 2-2. 流入 → 登録・体験（入口）

- **診断コンテンツを見た人が「自分もやってみる」までの距離が遠すぎる**。AI診断を受けるには Google 登録 → 本を複数冊登録 → メモ記入が必要。バイラル診断系は「30秒で自分の結果が出る」ことが必須条件。→ **未登録お試し診断**（好きな本を3〜5冊選ぶだけで簡易診断）が無いのはループ断絶の主因。
- ログインが **Google OAuth のみ**。日本のSNS流入なら LINE / Apple ログインの追加で登録CVRが大きく変わる。
- **他サービスからのインポートがない**（CSVエクスポートのみ、インポート無し）。読書メーター・ブクログからの乗り換え障壁が高く、インフルエンサー（＝既存サービスのヘビーユーザー）を取り込めない。
- 招待・リファラル機構が存在しない（invite/referral 関連のコード無し）。

### 2-3. 継続 → 再共有（ループの再点火）

- **メールが未接続**。React Email テンプレート（Welcome等）は `libs/resend/templates/` に存在するが、`resend.emails.send()` の呼び出しがどこにも無い。リテンション施策がゼロの状態。
- **プッシュ通知なし**。PWA manifest はあるが Service Worker / Web Push 未実装。いいね・コメント・フォローされた通知はアプリを開かないと気づけない。
- **ゲーミフィケーションなし**（ストリーク・バッジ・達成演出なし）。「◯冊読了」「◯日連続」のような達成イベントは、それ自体がシェアのトリガーになる。
- 月次の区切りコンテンツ（月間まとめ）がなく、シェア機会が年1回（Wrapped相当）しかない。

### 2-4. 計測

- 分析は Vercel Analytics（ページビュー）のみ。**ファネル（シェア→流入→登録→診断完了）や K係数を測る手段がない**。イベント計測（PostHog / GA4 等）なしにバイラル施策のPDCAは回せない。

## 3. 機能ロードマップ（インパクト × 実装コストで優先度付け）

### P0 — バイラルループを成立させる（最優先）

| # | 機能 | 概要 | 実装メモ |
|---|---|---|---|
| 1 | **シェア画像の直接共有・保存** | OGカードを画像としてダウンロード／Web Share API の `files` で直接共有。**9:16 縦型（Instagram Stories / TikTok）を追加** | `api/og.tsx` は流用可。`?format=story` で縦型レイアウト追加、フロントに「画像を保存」「画像でシェア」ボタン |
| 2 | **未登録お試しAI診断** | 本を3〜5冊選ぶ（MeiliSearch/外部API検索）だけで簡易性格診断 → 結果画像シェア → 「登録して精密診断」へ誘導 | 既存の `ai-summary` プロンプトの軽量版。レート制限必須（IP/anonymousId 単位）。LP `/shindan` として独立させる |
| 3 | **読了時シェアモーダル** | 読了ステータス登録直後に「読了カード」（表紙+感想+冊数）を出してワンタップ共有 | 感情ピークにトリガーを置く。既存 `BookDetailReadModal` の共有を昇格 |
| 4 | **イベント計測基盤** | PostHog（or GA4）導入。share_click / visit_from_share / signup / ai_summary_done をイベント化し、シェアURLに `?ref=` を付与 | これが無いと以降の施策の効果検証ができない |

### P1 — 流入の受け皿と再訪の仕組み

| # | 機能 | 概要 |
|---|---|---|
| 5 | **読書メーター/ブクログ CSVインポート** | 乗り換え障壁の解消。ヘビーユーザー（＝発信力のある読書家）の獲得に直結 |
| 6 | **月間まとめ + Wrapped の自動生成・配信** | 月初に「先月の読書まとめカード」を自動生成し通知。12月は年間Wrappedを全ユーザーに一斉解放 |
| 7 | **メール接続（Resend）** | 既存テンプレートの送信を実装。Welcome → 週間ダイジェスト（フォロー中の新着・もらったいいね）→ Wrapped通知 |
| 8 | **LINE / Threads / Instagram 共有導線 + LINEログイン** | `socialShare.ts` の共有先拡張。日本市場では LINE 共有の効果が大きい |
| 9 | **Web Push（PWA強化）** | Service Worker 追加。いいね・コメント・フォロー通知をプッシュ化 |

### P2 — 継続とネットワーク効果の強化

| # | 機能 | 概要 |
|---|---|---|
| 10 | **ストリーク・読了バッジ** | 「30日連続記録」「年間50冊」等の達成 → 達成カード自体をシェア対象に |
| 11 | **埋め込みウィジェット** | ブログ/noteに貼れる「私の本棚」embed。被リンク獲得とブランド露出 |
| 12 | **招待・リファラル** | 招待リンク経由の登録で双方に特典（AI診断回数の追加など。`aiSummaryUsage` のクォータ機構を流用可能） |
| 13 | **お題・企画機能** | 「今月のテーマ本」「読書会シート」など共同参加型コンテンツ |

## 4. 運用施策（機能開発と並行して行うこと）

### 4-1. 季節性キャンペーン（最重要運用）

- **12月: #kidokuWrapped**。年間読書まとめカードを全ユーザーに一斉解放し、X/Instagram 投稿キャンペーン。Spotify Wrapped が証明した「全員同時・比較したくなる・画像1枚」の型に完全に乗せる。
- **年始: 「今年は◯冊読む」宣言カード**。目標宣言はシェアされやすく、新規登録の動機にもなる。
- **文学賞シーズン**（本屋大賞4月・芥川直木賞1月/7月）: 受賞作の感想ページを `/discover` で特集し、検索流入とX言及の波に乗る。
- **読書週間（10-11月）・夏の課題図書**などの定例イベント連動。

### 4-2. SNS運用

- **X公式アカウント運用**: 公開メモ（`isPublicMemo`）の良質な感想を（許諾設計の上で）引用紹介。「今週読まれている本」ランキング（`popularBooks` のデータ）を定期投稿。
- **ハッシュタグ設計**: `#kidoku` に加え、診断結果タイプ別タグ（例: `#隠れ哲学者タイプ`）を診断カードに焼き込み、検索面での連鎖を作る。
- **読書系インフルエンサー施策**: P1-5 のインポート機能を武器に、読書メーター等のヘビーユーザーへ個別に移行サポートを提供。書評YouTuber/Vloggerに Wrapped カードを提供。

### 4-3. SEO・コンテンツ運用

- ISR 済みの `/books/[bookId]` を「書名 + 感想」のロングテールSEOの受け皿として強化（構造化データ `Book` / `Review` schema.org の追加、内部リンク: 同カテゴリの本・この本を読んだ人の本棚）。
- `/discover` の人気本・ランキングページを静的生成してインデックスさせる。

### 4-4. 計測・グロース運用

- 週次で追う指標: **シェア率**（診断完了→共有クリック）、**シェア経由流入数**、**流入→登録CVR**、**K係数**（1ユーザーが連れてくる新規数）、**AI診断完了率**、**W1リテンション**。
- 目安: K係数 0.2 超えで施策継続、0.5 超えでキャンペーン増強。診断→シェア率が 10% を切る場合は結果カードの文言・見た目を A/B。
- バズ発生時の備え: AI診断はレート制限とコスト上限（`aiSummaryUsage` の匿名ユーザー版）を事前に用意。Vercel/Cloud Run のスパイク時挙動を確認しておく。

## 5. 目指すバイラルループ（全体像）

```
[既存ユーザー] ─ 読了/診断/月間まとめ ─▶ シェアカード生成（画像・縦型対応）
      ▲                                        │ #kidoku + ref付きURL
      │                                        ▼
  継続利用 ◀─ メール/プッシュ/ストリーク    [X / Instagram / LINE]
      ▲                                        │
      │                                        ▼
  本登録・診断 ◀─ 登録(Google/LINE) ◀─ 未登録お試し診断（30秒で結果）
                                    ▲
                          インポートで乗り換え組も取り込む
```

## 6. まずやること（実装順の提案）

1. **計測基盤**（P0-4）— 以降すべての施策の前提。半日〜1日。
2. **読了時シェアモーダル + 画像保存**（P0-1, P0-3）— 既存OG生成の流用で小さく出せる。
3. **未登録お試し診断**（P0-2）— ループの入口。LPとして独立リリースし、それ自体を最初のバズ企画にする。
4. **Resend接続 + 月間まとめ**（P1-6, P1-7）— 12月の Wrapped キャンペーンに間に合わせるための布石。
