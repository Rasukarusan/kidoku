# Kidoku コアファン向け機能提案 — 「10年使い続けられる読書記録」へ

> バイラル成長（[VIRAL_GROWTH_ANALYSIS.md](./VIRAL_GROWTH_ANALYSIS.md)）とは逆方向の分析。新規獲得ではなく、**すでに使い込んでいる人の満足度と定着を最大化する**ための機能を考える。

## 1. コアファンは誰か

コードベースとデータモデルから見える、Kidokuを深く使う人の姿:

- **メモを書き込む人** — `memo` は MediumText、マークダウンエディタ・コピー機能・【期待】【感想】テンプレートまで整備済み。感想を「書く」ことが体験の中心にある
- **統計を眺める人** — 年別シート、月別グラフ、カテゴリ内訳、AI分析。自分のデータを見返すのが楽しい人
- **長く記録する人** — シートは年単位で積み上がり、`/total` で全期間を俯瞰する設計。数年スパンの利用が前提

つまりコアファンにとって Kidoku は SNS ではなく**「自分の読書人生の母艦」**。この方向に深掘りする。

## 2. 提案機能（テーマ別）

### テーマA: 記録の解像度を上げる — 「書く楽しさ」の深化

| # | 機能 | 内容 | 既存資産との接続 |
|---|---|---|---|
| A-1 | **読書ステータス** | 積読 / 読書中 / 読了 / 中断 の状態管理。現状は `finished` 日付の有無しかなく、「買ったけどまだ」「読みかけ」が表現できない | `books` に `status` enum + `startedAt` を追加。積読リストはコア読書家の必需品 |
| A-2 | **引用・抜き書き** | 本ごとに引用を複数記録（ページ番号・自分のコメント付き）。引用だけを横断で見返せる「引用ノート」ビュー | 新エンティティ `quotes`。MeiliSearch のインデックス対象に加えれば「あのフレーズどの本だっけ」が解決する |
| A-3 | **カスタムメモテンプレート** | 【期待】【感想】は現在ハードコード（`RegisterForm.tsx` 等）。ユーザーが自分の型（【学び】【引用】【アクション】など）を定義できるように | `template_books` の仕組みを「本の雛形」から「メモの雛形」へ拡張 |
| A-4 | **再読の記録** | 同じ本の2回目・3回目の読了を別レコードでなく履歴として持つ。「再読して感想がどう変わったか」はコア読書家の醍醐味 | `readings`（bookId, finished, memo）へ読了イベントを分離 |
| A-5 | **評価軸の拡張** | `impression`（VarChar(5)）の一軸評価に加え、任意の自分軸（没入度・難易度・再読したい度）を定義可能に | こだわりの強い人ほど独自の採点基準を持っている |

### テーマB: 蔵書と積読の管理 — 「本棚の母艦」化

| # | 機能 | 内容 |
|---|---|---|
| B-1 | **積読ダッシュボード** | A-1のステータスを前提に、積読数・積読日数・消化ペース・「この積読ペースだとあと◯年」を可視化。罪悪感ではなく楽しめるトーンで |
| B-2 | **所有形態・媒体の記録** | 紙 / Kindle / オーディオブック / 図書館で借りた、の区分。売却・貸出中フラグ。「この本持ってたっけ」を書店でスキャンして即答できる（既存バーコードスキャンの活用） |
| B-3 | **タグ・横断本棚** | シート（年）に直交する自由タグ。「哲学」「積読」「人に薦めたい」などでシートを跨いだ仮想本棚を作れる |
| B-4 | **シリーズ・著者フォロー** | 登録済み著者の新刊を openBD / 楽天ブックスAPIで検知して通知。「続きが出たら教えて」はコア読書家の切実なニーズ |

### テーマC: 自分データの分析 — 「読書人生の可視化」

| # | 機能 | 内容 |
|---|---|---|
| C-1 | **生涯統計ダッシュボード** | `/total` を拡張し、年別推移・累計冊数・著者ランキング・カテゴリの変遷（年ごとの興味の移り変わり）を1画面に。データは全て既存 |
| C-2 | **AI分析の時系列比較** | `AiSummaries` は年（シート）ごとに保存済み。「去年のあなた」と「今年のあなた」の diff をAIに語らせる。長く使うほど価値が増す＝コアファン専用の報酬 |
| C-3 | **「◯年前の今日」** | `finished` 日付から「3年前の今日、この本を読み終えていました」を通知/表示。過去メモとの再会は長期ユーザーだけが得られる体験 |
| C-4 | **年間レポートの私的版** | バズ用Wrappedカードではなく、自分だけでじっくり読む詳細レポート（月別・引用ベスト・メモ文字数・最長積読解消など）。PDF保存対応 |

### テーマD: データ主権 — 「一生ものだから逃げ道がある」信頼

| # | 機能 | 内容 |
|---|---|---|
| D-1 | **CSVインポート（復元・移行）** | エクスポートは既存（`api/export/csv.ts`）だがインポートがない。自分のバックアップを書き戻せない状態は「母艦」として致命的。読書メーター/ブクログ形式にも対応 |
| D-2 | **Markdown/Obsidian エクスポート** | メモのマークダウンコピーは1冊単位で実装済み。全蔵書を1冊=1ファイルのMarkdown(frontmatter付き)でzipエクスポートし、Obsidian/Notionユーザーの知的生産フローに接続する |
| D-3 | **個人用API/Webhook** | 読了イベントのWebhookやパーソナルアクセストークン。自分の記録を自分のツールに流したい層は、まさに10年使うコア層 |

### テーマE: 小さく深いつながり — 広場ではなく書斎

| # | 機能 | 内容 |
|---|---|---|
| E-1 | **公開範囲「フォロワー限定」** | 現在 `isPublicMemo` は全公開/非公開の2値。「親しい人にだけ見せたい」中間段階を追加（`Follow` モデルは既存なので判定は容易）。本音の感想が書けるようになり、メモの質と量が上がる |
| E-2 | **共有シート** | 夫婦の本棚・読書会・ゼミなど、少人数で1つのシートを共同編集。招待制のクローズドな空間はバズと対極だが定着効果が極めて高い |
| E-3 | **年間ベストの相互観賞** | `YearlyTopBook` は実装済み。フォロー相手の「今年のベスト」を年末にまとめて見られるページ。少人数の間で毎年の恒例行事になる |

## 3. 優先順位の提案

コアファンの「不満解消 → 報酬 → 信頼」の順で:

1. **A-1 読書ステータス（積読・読書中）** — 記録アプリとして最も基本的な欠落。B-1/B-2 の土台にもなる
2. **D-1 CSVインポート** — エクスポートだけあってインポートがないのはデータ主権の片翼。移行需要も拾える
3. **A-3 カスタムメモテンプレート** — 「書く人」への直接の報酬。実装も小さい（ハードコード文字列の設定化）
4. **C-1 生涯統計 + C-3 ◯年前の今日** — 追加データ不要、既存データの見せ方だけで長期ユーザーに報いる
5. **E-1 フォロワー限定公開** — メモの質を上げる構造的な一手。`isPublicMemo: Boolean` → `visibility: enum(private/followers/public)` のマイグレーション
6. **A-2 引用機能** — 新規エンティティで工数はあるが、「Kidokuにしかない自分の資産」を最も増やす機能

## 4. 判断基準（今後の機能検討用）

コアファン向け機能かどうかは次の3問で判定できる:

1. **使い込むほど価値が増えるか？**（データが溜まるほど嬉しい = C-2, C-3型）
2. **「自分の記録の母艦」としての信頼を高めるか？**（データ主権・表現力 = A, D型）
3. **静かで安心な場所か？**（数字を競わせない、広場に晒さない = E型）

逆に「新規ユーザーの目を引くか」「シェアされるか」はここでは評価しない。
